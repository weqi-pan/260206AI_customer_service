# 扫地机器人智能客服 Agent 后续优化计划

> 本文档已根据当前仓库实际状态重新评审和修订。
>
> 当前项目状态：可演示稳定版，并已完成第一阶段 SQLite 本地数据层。项目使用 Streamlit + LangChain/LangGraph + Chroma + SQLite + 本地样例 CSV + Provider，已经支持 RAG 自动初始化、确定性用户/城市/月度模拟、样例使用报告数据、基础多轮上下文和会话持久化。
>
> 本文档描述的是后续优化路线，不代表全部已经实现。

---

## 一、Review 结论

### 1.1 原方案中不合理的地方

1. **直接引入 MySQL 过重**
   - 当前项目是单机演示型应用，没有多用户登录、后台服务、部署拓扑和真实业务数据源。
   - 直接引入 MySQL 会增加安装、配置、连接池、初始化脚本和运维成本，不利于课程演示和快速运行。

2. **SQLite 示例和 MySQL 方案混用**
   - 原文参考示例使用 `sqlite3.connect()`，但后续方案又要求 MySQL 连接池。
   - 两种数据库的部署方式、SQL 方言、连接生命周期和测试方式不同，混在同一方案里容易误导实现。

3. **数据库表设计一次性过大**
   - `users`、`devices`、`inventory`、`device_usage_records`、`conversation_history` 同时落地，对当前项目来说范围过大。
   - 当前最紧迫的问题是替换 CSV 使用记录和稳定会话，而不是立刻做完整商品库存系统。

4. **工具数量扩张过快**
   - 原方案一次新增 5 个数据库工具，会让 Agent 工具选择更复杂。
   - 当前提示词已经包含 RAG、天气、用户、月份、报告工具，继续堆工具容易降低调用稳定性。

5. **缺少与现有 Provider 抽象衔接**
   - 当前项目已有 `agent/tools/providers.py`，用户、城市、月份、天气默认来自 demo Provider。
   - 后续用户系统和天气服务应该优先扩展 Provider，而不是把逻辑散落到工具函数里。

6. **对话记忆方案应先本地化，再数据库化**
   - 当前 Streamlit session 已能保存页面内历史，并传入 Agent。
   - 下一步更合理的是先做本地 SQLite 持久化，验证会话恢复，再考虑 MySQL。

### 1.2 保留的合理方向

1. 用户体系是必要的，但应先从“本地样例用户”开始。
2. 使用记录从 CSV 迁移到结构化存储是合理的。
3. 设备信息和库存查询可以作为第二阶段能力，而不是第一阶段必做。
4. 对话历史持久化有价值，但应先做轻量本地实现。
5. 后续若要真实部署，再切换到 MySQL/PostgreSQL 更合适。

## 二、推荐优化路线

### 阶段一：本地数据层稳定化（已完成）

目标：保持项目开箱可运行，不依赖外部数据库，同时把 CSV 和 session 内存状态升级为结构化本地数据。该阶段已落地到 `db/` 模块。

#### 2.1 技术选型

| 模块 | 推荐方案 | 原因 |
|------|----------|------|
| 本地数据库 | SQLite | Python 内置、零部署、适合演示和单机应用 |
| 数据访问 | 标准库 `sqlite3` + 简单 Repository | 当前项目规模小，无需 ORM |
| 初始化 | 启动时检测并初始化 | 保持克隆后可直接运行 |
| 数据迁移 | 从 `data/external/records.csv` 导入 | 兼容现有样例数据 |

#### 2.2 新增目录建议

```text
db/
├── __init__.py
├── connection.py        # SQLite 连接和路径管理
├── init_db.py           # 建表与样例数据导入
├── repositories.py      # 用户、使用记录、会话历史查询
└── schema.sql           # SQLite 建表脚本
```

#### 2.3 第一阶段表结构

只实现当前功能真正需要的表：

```sql
CREATE TABLE IF NOT EXISTS users (
    user_id TEXT PRIMARY KEY,
    username TEXT NOT NULL,
    city TEXT NOT NULL,
    phone TEXT,
    created_at TEXT DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE IF NOT EXISTS device_usage_records (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id TEXT NOT NULL,
    month TEXT NOT NULL,
    feature TEXT NOT NULL,
    efficiency TEXT NOT NULL,
    consumables TEXT NOT NULL,
    comparison TEXT NOT NULL,
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(user_id, month),
    FOREIGN KEY(user_id) REFERENCES users(user_id)
);

CREATE TABLE IF NOT EXISTS conversation_history (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id TEXT NOT NULL,
    user_id TEXT NOT NULL,
    role TEXT NOT NULL,
    content TEXT NOT NULL,
    created_at TEXT DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX IF NOT EXISTS idx_conversation_session
ON conversation_history(session_id, created_at);
```

#### 2.4 已调整的现有模块

1. `utils/generate_external_data.py`
   - 保留 CSV 解析能力。
   - SQLite 初始化阶段会从 CSV 导入样例记录，报告查询优先走 SQLite。

2. `agent/tools/providers.py`
   - 已新增 `SQLiteExternalProvider`。
   - `get_user_id()`、`get_user_location()` 从数据库读取默认用户。
   - demo Provider 保留，作为无数据库或测试场景 fallback。

3. `agent/tools/agent_tools.py`
   - `fetch_external_data(user_id, month)` 已改为通过 Repository 查询使用记录。
   - 工具接口保持不变，避免影响 Agent 注册和提示词。

4. `app.py`
   - 已生成并保存 `session_id`。
   - 用户消息和助手消息已写入 `conversation_history`。
   - 页面刷新后可从 SQLite 恢复最近 N 条历史。

#### 2.5 第一阶段验收标准

1. 删除本地数据库文件后启动应用，系统自动建表并导入样例数据。
2. 输入“给我生成我的使用报告”，能从 SQLite 查询用户 `1001` 的 `2026-05` 记录。
3. 刷新 Streamlit 页面后，最近对话能恢复。
4. 不安装 MySQL 也能完整演示。

## 三、阶段二：产品与库存能力扩展（可选）

目标：在数据层稳定后，再增加设备型号、价格、库存等客服能力。

### 3.1 表结构扩展

```sql
CREATE TABLE IF NOT EXISTS devices (
    model_id TEXT PRIMARY KEY,
    model_name TEXT NOT NULL,
    category TEXT NOT NULL,
    specs TEXT NOT NULL,
    description TEXT NOT NULL,
    price REAL NOT NULL,
    features TEXT NOT NULL
);

CREATE TABLE IF NOT EXISTS inventory (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    model_id TEXT NOT NULL,
    warehouse TEXT NOT NULL,
    stock INTEGER NOT NULL DEFAULT 0,
    updated_at TEXT DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(model_id, warehouse),
    FOREIGN KEY(model_id) REFERENCES devices(model_id)
);
```

### 3.2 新增工具建议

控制工具数量，优先只加 3 个：

| 工具 | 入参 | 用途 |
|------|------|------|
| `search_devices` | `keyword` | 按关键词搜索设备 |
| `query_device_info` | `model_id` | 查询型号详情 |
| `query_inventory` | `model_id` | 查询库存汇总 |

不建议单独新增 `check_stock_availability`，库存检查可以由 `query_inventory` 覆盖，减少工具选择复杂度。

### 3.3 验收标准

1. 用户询问“有哪些扫拖一体机器人”时，Agent 能调用 `search_devices`。
2. 用户询问“R1-Pro 怎么样”时，Agent 能调用 `query_device_info`。
3. 用户询问“R1-Pro 有没有库存”时，Agent 能调用 `query_inventory`。

## 四、阶段三：真实数据库部署（生产化时再做）

当项目从本地演示走向多人使用或服务端部署时，再考虑 MySQL/PostgreSQL。

### 4.1 触发条件

满足以下任一条件，再迁移到 MySQL/PostgreSQL：

1. 需要多用户同时访问。
2. 需要部署到服务器并长期保存数据。
3. 需要后台管理设备、库存、用户。
4. 需要和真实业务系统集成。

### 4.2 迁移策略

1. 保持 Repository 接口不变。
2. 新增 `MySQLRepository` 或 `PostgresRepository` 实现。
3. 通过配置切换：

```yaml
storage:
  provider: sqlite  # sqlite | mysql
  sqlite_path: data/app.db
```

4. MySQL 相关依赖只在该阶段加入：

```text
mysql-connector-python
```

### 4.3 不建议现在加入的内容

1. MySQL 连接池。
2. 独立数据库初始化 CLI。
3. 复杂 ORM。
4. 完整库存后台。
5. 大量数据库工具一次性注册给 Agent。

## 五、测试计划

### 5.1 单元测试

建议新增 `tests/` 目录，优先覆盖：

1. 配置加载和环境变量占位符解析。
2. SQLite 初始化和样例数据导入。
3. 使用记录查询成功和未命中场景。
4. conversation history 保存和按 session 恢复。
5. Provider fallback 行为。

### 5.2 集成测试

1. 删除本地 `data/app.db` 后启动，验证自动初始化。
2. 调用 `fetch_external_data("1001", "2026-05")`，验证返回结构化记录。
3. 连续两轮对话，验证第二轮能带上历史上下文。
4. 刷新页面，验证历史对话恢复。
5. 缺少 API Key 时，验证错误提示清晰。

### 5.3 手工验收问题

| 场景 | 示例问题 | 预期 |
|------|----------|------|
| RAG 问答 | 大户型适合什么扫地机器人？ | 调用 RAG 并回答选购建议 |
| 天气适配 | 上海潮湿天气能拖地吗？ | 调用天气工具并给出建议 |
| 报告生成 | 给我生成我的使用报告 | 查询使用记录并生成 Markdown 报告 |

## 六、推荐实施顺序

1. **SQLite 数据层**
   - 建表、初始化、CSV 导入、Repository 查询。
   - 状态：已完成。

2. **Provider 接入**
   - 用 SQLite Provider 替换当前 demo Provider 的用户和城市来源。
   - 状态：已完成。

3. **会话持久化**
   - Streamlit session 保持即时状态，SQLite 负责跨刷新恢复。
   - 状态：已完成。

4. **扩展设备和库存工具**
   - 在第一阶段稳定后再增加工具，避免一次性扩大 Agent 行为面。
   - 状态：待实现。

## 七、当前结论

原方案中“数据库化、用户体系、对话记忆、设备库存”的方向是合理的，但直接以 MySQL 作为第一步不适合当前项目。

更合理的路线是：

1. 当前阶段已经保持可演示稳定，并完成 SQLite 本地结构化数据和对话记忆。
2. 下一阶段根据需要扩展设备与库存能力。
3. 最后在真实部署场景下迁移到 MySQL/PostgreSQL。
